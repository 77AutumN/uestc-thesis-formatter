# D23 — refs_to_bib 漏标准/会议/报告类型

## 症状
CASE-A v6: references_raw.txt 30 条, 但 ref.bib 只 26 条 `@` 条目 + 4 条 `% WARNING: 未知文献类型`.
客户报"参考文献少了一条" — 实际历史已被 case-private 手补几条非 MJDN 类型, 还差一条标准条目.

## 根因
`refs_to_bib.py` main loop 第 372-381 行只处理 `[M]/[J]/[D]/[N]` 4 类, 其它类型进 else 走 "未知文献类型" warning 而**不写入 ref.bib**.

实际客户原稿常见类型:
- `[C]` 会议论文: Author A B...[C]. Conference, 2005, 3-7.
- `[S]` 标准: 示例标准化技术委员会(SAC/TC 999). GB/T 99999-2023[S]. 北京: 示例标准出版社, 2023.
- `[R]` 报告 / `[P]` 专利 / `[Z]` 其它: 不那么常见但应支持

## 修复 (shared)
`scripts/refs_to_bib.py`:
1. 新增 `parse_proceedings(text, citekey)` → `@inproceedings{}` 含 booktitle/year/pages
2. 新增 `parse_standard(text, citekey)` → `@misc{}` 含 howpublished/note=`[S] 标准`
3. 新增 `parse_report(text, citekey, type_marker)` → 通用 `@misc` 处理 [R][P][Z]
4. main 增 5 个 `elif '[C]'/'[S]'/'[R]'/'[P]'/'[Z]'` 分支

## CASE-A expansion (二次命中)
**症状**: CASE-A references_raw.txt 9 条, 但 ref.bib 仅 1 条 (8 个 PARSE_ERROR + 1 SKIP). 全因 D23 原版只支持 `[C].` 形式, 不支持远比它常见的 `[C]//` 形式.

**根因**: `parse_proceedings` line 327 `text.split('[C].')` 只覆盖 `Author. Title[C]. Conf, Year, Pages.` 形式 (CASE-A 的 Tummala 个例). GB/T 7714 §8.4.2 标准会议格式是:
```
[N] Author. Title[C]//<Conference name>. <Year>: <Pages>.
```
即 `[C]//` 双斜杠 + `<Year>: <Pages>` 冒号分隔. 这是 CVPR / ICCV / NeurIPS / IAPR / ECCV 等顶会全部通用的格式.

**修复 (shared, CASE-A)**: `parse_proceedings`
- 优先 `text.split('[C]//', 1)`, 兜底 `text.split('[C].')`
- 末尾页码 regex 字符类增加 `:` `：`: `r'(\d{4})\s*[:,：，]\s*([\d\-+]+)\s*$'`
- 增加无 pages 兜底分支: `r'(\d{4})\s*\.?\s*$'` → 仅取 year, pages 占位

**测试**: 3 个新 fixture 在 `tests/test_refs_to_bib_parsers.py`:
- `test_d23_expansion_proceedings_double_slash` (主路径)
- `test_d23_expansion_proceedings_year_only` (无 pages 兜底)
- `test_d23_expansion_proceedings_falls_back_to_dot` (D23 原版未破坏)

## 当前 case-private (CASE-A v7)
跑修后的 refs_to_bib 一遍, 30/30 全成功, ref.bib 含 r2820 + 3 条之前手补的也变规范 entry.

## 如何识别
- ref.bib 头部注释 "共 N 条" vs grep `^@` 实际 @ 条数 不一致
- ref.bib 含 `% WARNING: 未知文献类型` 注释行
- `cite_map.json` size > ref.bib `@` 条数
